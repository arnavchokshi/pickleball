## Current Spine Reality

`threed/racketsport/orchestrator.py` currently registers real tracking by default and keeps a manual precomputed mode:

| runner | real_model | source_mode | notes |
|---|---:|---|---|
| `ManualCalibrationRunner` | false | `manual_sidecar` | Writes calibration, zones, and net artifacts. Uses real geometry/solvePnP where available, but still depends on reviewed manual sidecar inputs for prototype runs. |
| `RealYOLO26BoTSORTReIDTrackingRunner` | true | `yolo26m_botsort_reid` | Verifies the manifest `yolo26m` checkpoint sha256 before `YOLO(...)`, then invokes Ultralytics `model.track(..., tracker=botsort_reid.yaml)` with ReID enabled and writes `tracks.json`. |
| `PrecomputedTrackingRunner` | false | `precomputed_detections` | Explicit manual/prototype mode only (`--tracking-mode precomputed`). Converts existing `detections.json` into `tracks.json`. It does not invoke YOLO26m, BoT-SORT, or ReID. |
| `BodyStageRunner` | true | `fast_sam_3d_body` | Writes `body_compute_execution.json`, consumes `frame_compute_plan.json` `deep_mesh_windows` when present, verifies `fast_sam_3d_body_dinov3`, `sam_3d_body_mhr_model`, `moge_2_vitl_normal`, and `yolo26m` sha256s before loading, invokes Fast SAM-3D-Body on scheduled tracked player boxes, converts camera outputs through `court_calibration.json`, and writes `smpl_motion.json` with world joints/mesh vertices plus preview `skeleton3d.json`. |
| `BallStageRunner` | false | `no_click_fusion_temporal_vball100_localtraj` | Consumes a strict no-click prototype ball track only when it is supplied as an explicit source, colocated input/run artifact, selected-track artifact, or explicit prototype-root fallback opt-in, then writes `ball_track.json`; machine cue fusion still fails closed without complete audio+wrist+ball cue triplets. | `RacketStageRunner` | false | `explicit_four_corner_candidates_pnp_ippe` | Consumes schema-validated `racket_candidates.json` with explicit four-corner paddle candidates, solves PnP-IPPE, writes `racket_pose.json`, and fails closed when the artifact is missing/malformed, candidates are ambiguous, or reprojection is above threshold. `run_pipeline --stage e2e` still stops honestly at the first missing or unavailable real runtime. Real tracking fails loudly if the source video, manifest checkpoint, checksum, tracker config, or Ultralytics runtime is missing. BODY now writes a schedule manifest before model checks and fails loudly if the H100 Fast SAM-3D-Body repo, required checkpoints, checksum verification, or adaptive schedule is unavailable. After a run, the orchestrator regenerates unprotected best-effort review helpers (`frame_compute_plan.json`, `body_compute_execution.json`, `virtual_world.json`) from current artifacts when possible instead of trusting stale helper files, while preserving successful stage-owned BODY plan/execution/readiness artifacts; it embeds the artifact-plus-semantic readiness report in `pipeline_run.json`, and downgrades an otherwise passing run to `blocked` when readiness is semantically not ready. `pipeline_readiness_e2e.json` is packet-indexed as an artifact-plus-semantic readiness report; regenerate it after contact/BODY artifact changes before treating its blocker text as current. `contact_window_candidates.json` is intentionally separate from canonical `contact_windows.json`; it exists to help review uncertain contact hints and is not consumed by BODY. The current accepted-four canonical `contact_windows.json` files came from explicit human review inputs, not complete machine cue triplets, and remain prototype evidence rather than BALL verification. The BODY contract preserves `mesh_vertices_world` when the runtime returns vertices. Scheduled H100 BODY stage runs have passed for Burlington, Wolverine, and Indoor, while Outdoor remains zero scheduled; BODY still needs labeled world-MPJPE gates before promotion. The furthest locally verified prototype vertical slice remains calibration plus precomputed tracking.

## Gate Labels

Current evaluator gates for body, foot/physics, metrics, replay, and e2e are presence, artifact-completeness, or structural artifact checks. Replay/e2e GLB checks parse GLB 2.0 header/chunk/glTF JSON structure, but they do not prove visual correctness, animation quality, compression, USDZ export, or accuracy. Tracking, ball/events, and racket now add `label_check.*` metrics only when reviewed labels are authoritative; prototype draft or `not_ground_truth` labels are reported as `not_measured`. Racket artifact player/frame/contact counts are recorded as `artifact_check.*` context and cannot pass Phase 6 without reference pose labels. Presence checks, artifact checks, sparse prototype benchmarks, and unreviewed label checks do not count toward `VERIFIED`.

Shot classification gates now report reviewed-label macro-F1, top-2 accuracy, missing-prediction coverage, per-class precision/recall/F1, unknown/gated rates, and confusion when authoritative labels exist. The current calibration bucket helper lives in `shot_event_eval.py`, but the phase gate does not yet surface a dedicated calibration metric. Current external checks are insufficient for promotion: THETIS 60-clip family accuracy is 51.7% overall / 66.7% top-2 with `serve` 0/15 and `overhead` 0/5, while OpenSportsLab 100-event broad accuracy is 81.0% but `serve` is 0/19. Broad swing or FH/BH success cannot be used to claim serve, overhead, or full shot-taxonomy readiness.

The target accuracy gates remain:

| area | real gate required before VERIFIED |
|---|---|
| body | world-MPJPE target, roughly 50-70 mm on labeled evaluation data |
| ball/events | ball F1 at least 0.90 and contact timing within plus/minus 2 frames |
| racket | face-angle error at most 5 degrees against ArUco/GT |
| foot/physics | foot-slide at most 3 mm and zero penetration |
| metrics | metric accuracy against manual review plus confidence calibration |
| replay/e2e | generated assets load correctly and end-to-end SLA/quality gates pass on real clips |

## Real Smokes That Exist But Do Not Promote Status

- YOLO26m plus BoT-SORT-ReID is now a registered tracking `StageRunner`, but the real H100 clip smoke and labeled IDF1/spectator-rejection gate have not passed yet.
- Person-tracking benchmarks can now emit candidate-local `frame_compute_plan.json` and `body_compute_execution.json` from each candidate `tracks.json`, which makes adaptive BODY scheduling impact visible per tracker variant. This is a review/audit output only and does not promote TRK or BODY.
- TrackNetV3 has real H100 smoke windows and an adapter that can run the official predictor when the external runtime exists, but BALL lacks full-clip contact-window generation and label F1/timing gates.
- The current strict no-click ball review artifact is `ball_track_fusion_temporal_vball100_localtraj.json` under each accepted clip's `tracknet_smoke_0000_0010/` directory. It improved the four-clip benchmark from 8 teleports to 0 and hidden false positives from 0.425 to 0.294 versus the looser fusion track, with hit recall dropping from 0.563 to 0.509. This is useful for prototype review but does not promote BALL.
- Accepted-four `ball_inflections.json` artifacts surface review-only court-plane trajectory-turn cues from `virtual_world.json`; after suppression the counts are Burlington 13, Wolverine 9, Outdoor webcam 22, and Indoor doubles 12. Accepted-four `audio_onsets.json` and `wrist_velocity_peaks.json` artifacts are packet-indexed too: Outdoor has 4 review-only audio energy peaks, Indoor has 1, Burlington/Wolverine are blocked by `no_audio_stream`, Burlington/Wolverine/Indoor BODY skeletons and a provisional SAM-3D-Body/MHR70 semantic map exist but the wrist-cue builder has not consumed that adapter/default explicit wrist indices yet, and Outdoor lacks `skeleton3d.json`. `build_contact_windows_from_cues.py` can fuse these cue artifacts into canonical `contact_windows.json` when all three source families temporally agree, but the current promoted accepted-four windows came from explicit human review inputs. This still does not promote BALL until real label F1/contact-timing gates pass.
- Fast SAM-3D-Body has a Burlington frame probe with mesh/keypoint tensors, and BODY now has a registered checksum-gated StageRunner plus camera-to-court conversion code that keeps world joints and mesh vertices in `smpl_motion.json`. A 2026-06-28 scheduled H100 BODY spine smoke produced schema-valid BODY artifacts for Burlington, Wolverine, and Indoor doubles, with 18,439 mesh vertices per player frame. Current `body_mesh_readiness.json` audits report Burlington `world_mesh_required_available_unverified` with 9 mesh player-frames, Wolverine `world_mesh_required_available_unverified` with 12, Indoor doubles `world_mesh_required_available_unverified` with 6, and Outdoor webcam `no_world_mesh_requested`. Current `body_compute_execution.json` schedules Burlington 3 frames / 9 player-frames, Wolverine 4 / 12, Indoor doubles 2 / 6, and Outdoor 0 / 0. This does not promote BODY: local runs still lack `/opt/fast-sam-3d-body` and `/workspace/checkpoints/body4d/...`, Outdoor has no scheduled BODY frames, and the world-MPJPE gate has not passed.
- Draft YOLO26m racket-box labels have been converted into strict `racket_candidates.json` artifacts for the accepted four clips and surfaced in the review packet with H.264 candidate overlay videos. The overlays explicitly scale 960x540 label coordinates to 1920x1080 video coordinates and now render all current candidate frames; Outdoor webcam and Indoor doubles use recovered 30-second source clips under `racket_candidates/` for full candidate coverage. The default PnP runner rejected the candidates as ambiguous, so no canonical `racket_pose.json` is claimed. Separate `racket_pose_preview.json`, `virtual_world_paddle_preview.json`, and Three.js `virtual_world_paddle_preview.html` artifacts show the ambiguous paddle meshes and approximate court-plane ball paths for review only.
- Accepted-four `racket_promotion_audit.json` artifacts now prove the preview path has not promoted box-derived frames into canonical `racket_pose.json`: all four are `safe_preview_only`, `canonical_racket_pose_present=false`, `promoted_pose_frame_count=0`, and `unsafe_promoted_frame_count=0`. The packet also includes one global `racket_model_runtime_readiness.json` report; it is CPU-only, claims no model has run, and is blocked because SAM 3, DINO-X/Grounded-SAM2, FoundationPose, GigaPose, FoundPose, paddle CAD/reference assets, and GT/reference labels are not declared ready.
