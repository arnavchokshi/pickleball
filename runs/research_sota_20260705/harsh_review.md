# Roadmap Review: Final Fix List

## Scope and method
This synthesis dedupes and ranks findings from an adversarial review pass over NORTH_STAR_ROADMAP.md, EDGE_PLAYBOOK.md, and their cross-references into CAPABILITIES.md, BUILD_CHECKLIST.md, RESET_HANDOFF_20260705.md, and the underlying codebase. Findings were graded across six lenses (technical-correctness B, already-built-vs-repo B+, self-contained B-, internal-consistency C+, feasibility-gates C+, production-readiness D+) and each severe finding was independently re-verified against live repo state before inclusion here.

## What's solid
The factual backbone of the roadmap holds up well. Cited research (Lindsey/Steyn aerodynamics numbers, the SMART/SMPLest-X paper, BOP HANDAL leaderboard figures, Talking Tennis's 317-output fabrication-audit claim) mostly checks out against primary sources on direct fetch. Claims about what's "already built" in the repo (fused paddle estimator, IDF1 scorer, camera fingerprinting) are almost always literally true in the code, even when the roadmap text misattributes *which file* they live in. This is a genuinely research-grounded, engineering-literate document — the defects are concentrated in execution readiness and internal bookkeeping, not in the underlying technical claims.

## The critical blocker
P4-2 calls itself "STAGED — one command," but the cited training script path doesn't exist anywhere in git. Investigation found the script *was* written, correctly, per spec — just committed under a sibling lane's handoff directory instead of its documented path. This is the single highest-priority fix: a `git mv` and two comment corrections turn a broken "one command" claim into an actually-true one. It's illustrative of a broader pattern: several "done" or "staged" claims in this doc set describe work that happened, just not quite where or how the doc says it did.

## High-severity cluster: things that will fail on first contact
Three findings cluster around actions that fail immediately if executed literally: P4-1's "Land Wave A" patch fails `git apply --check` on four bogus self-referential symlink hunks plus real content conflicts, and its documented cherry-pick fallback also fails on a dirty BUILD_CHECKLIST.md and an unmentioned web/replay/package.json conflict. Separately, the master roadmap and its companion playbook are themselves untracked in git, directly contradicting the "fully pushed" claim that RESET_HANDOFF's own cold-start procedure relies on. These are the kind of defects that turn "read the roadmap and go" into "read the roadmap, hit an error, then reverse-engineer the actual state" — exactly the failure mode a roadmap exists to prevent.

A second high-severity cluster is about internal consistency between the two companion documents: EDGE_PLAYBOOK mints task IDs (P4-0, P1-3(e)) and asserts in its own "roadmap-deltas" section that these were applied to NORTH_STAR_ROADMAP.md — but in both cases, the target document was never actually edited. This means EDGE_PLAYBOOK's delta list, which exists specifically to be a trustworthy changelog, is not reliable, and any agent trusting it over the primary doc will reason about phantom task IDs.

A third cluster involves a gate that cannot mathematically be met by its own currently-booked plan: Phase 5's "≤1× video duration" target is 30-48x away from what the lane's own evidence file documents as the achievable ceiling of every currently-scheduled speed lever, with no task anywhere proposing the order-of-magnitude BODY-runtime redesign that would be needed to close that gap. A downstream productization claim (P7-3's "≤10min/game beats pb.vision's 30min") inherits this same unmoored-math problem.

## Production-readiness: the weakest pillar
This is where the roadmap is thinnest. The product design explicitly commits to capturing and persistently storing biometric data — ReID embedding galleries, frozen body-shape betas — for non-owner "friends," which is the explicitly stated near-term plan, yet the only legal-review gate anywhere is a patent-claim check gated on "commercial launch" (a milestone that postdates friend onboarding). There is no consent mechanism, no retention policy, no delete-account flow. Separately, per-clip QA (P5-6) only fires after the full 17-stage GPU pipeline has already run — there's no cheap pre-flight gate to reject wrong-sport, corrupted, or badly-oriented video before burning full pipeline cost, and orientation detection is a hardcoded stub today. Cost metering (P5-4) is scoped to GPU-seconds only, omitting storage, LLM API cost, and human-review labor, yet that partial figure is used to assert whole-product margin confidence in P7-3. And the server code that exists today (single-process, in-memory job store, ephemeral /tmp storage, zero client auth) is treated as a "seed" rather than decomposed into the auth/storage/queue sub-tasks it actually needs.

## Circular and under-specified dependencies
Phase 0's own exit gate requires the ball-reflection factor to report "non-dormant," but three separate passages confirm that factor cannot activate until a Phase 1 task lands — a genuine forward dependency baked into the phase framed as running before everything else. Phase F's global-fusion optimizer also drops two physically necessary constraints that earlier phases explicitly promised it would carry forward (net-as-occluder, hand-paddle grip consistency), and its camera-trajectory input names only ARKit with no stated fallback for the many clips that won't have valid ARKit sidecars.

## Recommendation
Fix the critical/high items in the order given — most are cheap (a `git mv`, a missing checklist bullet, a doc commit) relative to their blast radius. The Phase 5 gate and the privacy/retention gaps deserve real design time before being treated as scoped work. Everything below rank ~20 is legitimate but lower-leverage documentation hygiene that can be batched into a single cleanup pass.
