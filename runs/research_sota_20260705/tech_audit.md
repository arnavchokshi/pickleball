# NORTH_STAR 7-Pillar Audit Synthesis

All 7 pillars returned **keep-with-modifications** except **body**, whose audit is a stub (verdict_reason literally says "test" — no real findings, no repo checks; treat as not-yet-audited, not as a clean bill of health). Every other pillar's finding below is grounded in a specific repo_reality_check (file/line/grep/number), not opinion.

---

## 1. ball-detector — keep-with-modifications
- **Fusion is a measured liability, not an asset, at the consensus layer.** Raw 4-detector 2D spatial-consensus fusion has hidden-FP 0.3493 — 5.5x worse than the single zero-shot WASB-tennis detector alone (0.063). The full ensemble+physics chain still loses to that single detector on held-out F1 (0.6969 vs 0.7248) after 4 consecutive internal→held-out inversions (`runs/lanes/ball_tracking_long_run_STATUS.md:519-522`). Union recall ceiling (0.8793) is real value — but at candidate generation, not at the voting/fusion step.
- **Two BOM-table claims are ahead of the repo.** `third_party/TrackNetV4` is untracked raw vendor code — zero motion-prompt reimplementation exists outside third_party/, contradicting EDGE_PLAYBOOK.md:374's "re-implemented" claim. WASB integration is vanilla zero-shot weights + a naive nearest-neighbor gate (`_select_best`), not the paper's HLSM training or Kalman-style inference filter — no HLSM code exists anywhere.
- **Schema gap:** `BallFrame.visible` is a plain bool; TOTNet's 4-level visibility-weighted BCE cannot be adopted as a pure loss swap — BallFrame + CVAT labeling need a 4-level occlusion field first (unstated prerequisite in P1-1).
- Repo surprise (positive): TOTNet's actual shipped recipe is fully extractable and portable onto our existing WASB HRNet backbone with no redesign.

## 2. ball-3d — keep-with-modifications
- **Solver is mischaracterized.** Not "IRLS" — it's bounded robust NLS (scipy TRF+Huber) wrapped by a genuine two-point BVP shooting solve (Newton+FD-Jacobian+backtracking), more capable than the plan's description implies.
- **Magnus/lift is 0% built**, not a "closed-form constant swap" as P0-7's text implies. Only drag (Cd=0.33/0.45) is seeded; zero Cl term, zero spin state, and no P0-7 simulator file exists anywhere in the repo.
- **P1-4a itself is only PARTIAL** (`runs/lanes/ball_p3a_bvp_anchor_first_20260705/report.json`): D.3(b) FAIL (5 baseline-good intervals regressed on reselection), internal-val F1 never rerun. Planning a symmetric bake-off against a not-yet-built learned-lift track (P1-4b) on 3-4 clips is premature given this project's own repeated lesson that small-sample fine-tunes overfit.
- Of the 4 "unique anchors" cited as differentiators (audio contact times, ball-diameter depth, blur-speedometer, ARKit pose), only the first two are actually wired into the solver's cost function today. Blur-speed (H13) has zero code and needs an unscheduled physical drop-test/radar session; ARKit pose only reaches the solver indirectly via calibration.

## 3. body — keep (stub audit, not evaluated)
No repo checks, no findings, verdict_reason is a placeholder ("test"). Treat this pillar as **not yet audited** — flag to the manager to re-run before relying on it.

## 4. paddle — keep-with-modifications
- **Fingers/palm-normal are near-useless alone** — palm-only IoU 0.065, *worse* than the old crude proxy (0.111); detector boxes supply ~100% of the measured accuracy gain (`RESET_HANDOFF_20260705.md:198-199`). WiLoR (already-queued P3-3) is the repo's own named fix — not a new keypoint/mask/homography pipeline.
- **Ball-reflection-at-contact is dormant but cheap to unlock**, and the plan's own architecture doc calls it "the strongest orientation evidence in the whole system" — blocked only on ball-chain 3D velocities (a cross-pillar dependency — see Contradictions below).
- **Pixel-geometry math** (using RacketVision's own published keypoint noise) shows 5-keypoint PnP is only clearly favorable at <6-8m on sharp frames — worst exactly at swing/impact, the highest-value coaching moment. Should be sparse/gated, not a co-equal continuous channel.
- **NORTH_STAR_ROADMAP.md's ForeHOI claim is factually wrong**: verified against the paper/GitHub directly — full pipeline runs "within one minute," not "<1s" (1-2 orders of magnitude off), and its 6-DoF pose piece is a bolt-on to FoundationPose, inheriting FoundationPose's own documented HANDAL weakness (0.04-0.26 AP on thin handheld objects) plus a separate NVIDIA license needing legal review. Demote to backlog SPIKE.
- The 544/544 "IPPE-ambiguous" finding used to argue against keypoint-PnP is really a bbox-corner correspondence failure, not the classical near-fronto ambiguity — a real corner-keypoint detector sidesteps that specific failure (but still faces the range/blur SNR problem quantified above).

## 5. court-placement — keep-with-modifications
- **The headline "19.8px p95 calibration floor" conflates two different error sources on two different code paths.** It's specifically the Burlington/Wolverine GEO auto-find number (a discrete line-to-semantics assignment/lock-on bug); the owner-known-court (v1) path is already 1.6-4x tighter (12.3px reviewed metric15, ~4-7px manual Outdoor). The real v1-relevant floor is IMG_1605's ~53px mean edge-of-frame distortion residual — a different problem, fixed by P4-4 (k1/k2), not P4-2/P4-3.
- **"App already records ARKit pose/gravity" is false.** Zero `import ARKit` anywhere in `ios/`; `ARKitSetupPassSidecar` is schema-only, never constructed in production. Only CoreMotion gravity is real. PoseGravity (cited as a load-bearing plan leg) has nothing to consume until P0-10 (real ARKit session integration) actually ships — and it appears exactly once in the roadmap with no assigned P4-x task.
- **net_plane.py already does linear post-to-center sag interpolation** with regulation constants — the roadmap's "today net_plane.json is a single plane" claim is imprecise. The real gap is regulation-default heights vs per-court tape measurements — a 1-line data fix, not the full P4-6a/b segmentation+catenary build the plan proposes.
- P4-0 (CourtProfile registry, the plan's own first-priority bet) is 0% built in code today, despite being described as "ready to build in days."

## 6. fusion — keep-with-modifications
- **Architectural diagnosis is directly confirmed in code**: `virtual_world.py`'s `_court/_players/_ball/_paddles` each pull from separate artifacts with zero cross-terms — Phase F's premise is solid.
- **ARKit per-frame camera trajectory is a zero-coverage happy path today.** Checked 20 real `capture_sidecar.json` files across every recent run lane — `arkit_camera_pose` and `ondevice_pose_track` are `None` in all 20. PF-2 will run in the harder P2-1 fallback branch (itself still "to build") on 100% of real data until P0-10 physically lands. (Independently confirms court-placement's ARKit finding — see Contradictions.)
- **Ceres/GTSAM is the wrong solver pick.** Neither is installed or imported anywhere in this environment/codebase; every existing optimizer here (`ball_arc_solver.py`, `racket6dof.py`, `court_calibration.py`) uses `scipy.optimize.least_squares` or `cv2.solvePnP`. JOSH itself (the cited precedent, verified via WebFetch) uses a plain PyTorch gradient-based optimizer, not Ceres/GTSAM.
- JOSH's ablation number (contact-scale loss alone: 314.3→148.9mm) is being cited as if it validates a decoupled post-hoc PF-1 correction — it's actually a within-one-continuous-optimizer loss-term ablation, architecturally different from a discrete post-hoc heuristic snap.
- `foot_pin.py` already implements the exact bounded-correction pattern (`max_correction_m`) PF-1 should reuse for both ball-paddle snap and foot-ground clamp.

## 7. speed-arch — keep-with-modifications
- **Diagnosis correct**: BODY is 97-98% of E2E wall; ~810s of ~925s stage time is plumbing (handoff 376-489s, payload assembly 343s) vs only 18s of actual GPU inference.
- **P5-1's "489→under-40s partially proven" overstates what happened.** The one live attempt at binary/mmap transport was a *measured regression* (1057.4s→1300.7s total wall; handoff itself went 376→489s, i.e. worse), reverted to pickle. Only round-trip correctness was proven, not the speed win.
- P5-3 (TensorRT) is real but touches <3% of E2E wall — should be scoped to WASB+YOLO26 only; SAM-3D-Body's ViT/MHR backbone is the highest conversion-risk target for a return that doesn't even hit the bottleneck.
- **P5-7's warm-worker-pool addressable ceiling is only 65.6s/clip** (model_load 23.3s + compile_warmup 42.3s), not the 489s handoff or 343s payload-assembly costs — caps at ~15-20% of a post-P5-1 floor, not the order-of-magnitude the spec implies. Must be written into the spec explicitly.

- [trick] Use the pipeline's existing per-subsystem CONFIDENCE fields as the factor weights directly, rather than hand-tuning new uncertainty parameters for PF-2.: event_fusion.py, racket6dof.py (SE3PoseConfidence), and the ball chain all already emit per-frame/per-event confidence scores.
