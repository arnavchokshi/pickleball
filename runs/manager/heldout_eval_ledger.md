**Purpose.** `outdoor_webcam_iynbd_1500_long_high_baseline` (`cvat_upload/03_outdoor_webcam_iynbd_1500_long_high_baseline_frames_0000_1150.mp4`, 770 reviewed visible ball labels + 381 hidden labels at `runs/cvat_imports/2026_06_30/outdoor_webcam_iynbd_1500_long_high_baseline/reviewed_boxes.json`) and `indoor_doubles_fwuks_0500_long_mid_baseline` (CVAT export pending as of 2026-07-01, tracked in `BUILD_CHECKLIST.md`; **landed 2026-07-02, see append at end of file**) are the **strict held-out eval clips** across every stage -- Burlington and Wolverine are used as train/internal-val only. Every session (Claude or Codex) that scores anything against a strict clip's CVAT labels, sweeps a threshold against it, or uses its frames/labels as a training or checkpoint-selection signal, is spending down the one asset that lets us claim an honest number for that stage. Multiple uncoordinated single checks across parallel sessions collectively mine the test set even when no single session intended to. This file is the append-only record of every such touch, across all stages, so a manager can see the cumulative exposure and enforce pre-commitment going forward.

**Enforcement mechanism.** This ledger is the audit trail for legitimate held-out *evaluation* touches (final scoring against a strict clip's labels is expected and allowed, exactly once per pre-committed candidate). It does **not** by itself prevent a strict clip from leaking into *training* or *validation-during-fitting* inputs -- that is now a code-level property, not just a documented process. `threed/racketsport/eval_guard.py`'s `assert_not_training_on_eval_clip(...)` is the enforcement mechanism: it is wired into every known training/dataset-build entry point (`scripts/racketsport/train_court_keypoint_heatmap.py`, `scripts/racketsport/train_person_osnet_reid.py`, `scripts/racketsport/build_ball_tracknet_cvat_dataset.py`, `scripts/racketsport/prepare_tracknetv3_finetune_dataset.py`, `scripts/racketsport/export_cvat_detection_yolo_dataset.py`, `scripts/racketsport/export_person_yolo_dataset.py`) and raises `EvalClipLeakError`, naming the violating path, before any of those entry points can silently ingest Outdoor or Indoor. Burlington/Wolverine may only be used as a validation-during-fitting signal when a caller explicitly opts in via `allow_internal_val=True`; Outdoor/Indoor have no override at all. Rows in this ledger where a strict clip was used as training/checkpoint-selection input (e.g. row 6, row 13 below) predate that guard and are exactly the incidents it exists to make structurally impossible going forward. The guard does not gate final, pre-committed, once-only *scoring* runs -- that discipline is still enforced by the POLICY section below and by this ledger.

**Columns.** *Pre-committed?* = was the candidate checkpoint **and** the threshold value fixed using only non-held-out (train-clip / internal-val) evidence, written down, before the strict clip was touched. `NO` includes any run that swept multiple thresholds directly against a strict clip's labels and reported the best one, or that used its frames/labels as a training/validation signal during model fitting. `N/A (pre-ledger baseline)` = a single fixed-config candidate run that predates this ledger and was not the product of threshold-shopping on the strict clip, but also was never explicitly pre-registered anywhere.

Metrics are micro F1@20px / recall@20px / precision@20px / hidden-FP rate / teleport count against the full 770-label Outdoor set unless noted otherwise. The M1 BALL gate is F1@20 >= 0.90, recall@20 >= 0.75, hidden-FP rate <= 0.05 (`threed/racketsport/ball_detector_gate.py`). No candidate below has ever passed it.

3. Each session gets **exactly one** run per strict clip per pre-committed candidate. If a run fails to produce a usable result (crash, OOM, wrong config), that is a new row with a note explaining the retry — not silent overwriting.
4. **When in doubt, the manager decides.** If a session is unsure whether a planned action counts as "touching" a strict clip (e.g., using its frames for anything other than final scoring), stop and ask before running rather than logging it after the fact.
5. This file is append-only. Do not edit or delete prior rows; add corrections as new rows that reference the row being corrected.
6. Indoor becomes a second strict clip the moment its CVAT export lands (see `DATA-1 Indoor resolution` in `BUILD_CHECKLIST.md`); from that point on, every rule above applies to Indoor exactly as it already applies to Outdoor, and `threed/racketsport/eval_guard.py`'s registry already enforces this in code (Indoor has been a registered strict-holdout clip id since the guard was added, independent of whether its labels exist yet).
